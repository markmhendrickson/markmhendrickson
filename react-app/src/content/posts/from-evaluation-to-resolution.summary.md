- Agent-led evaluation via neotoma.io/evaluate told users whether the product fit. The new issues subsystem (v0.12.0/v0.12.1) closes the other half of the loop: when the user's agent hits a bug, it files the report through MCP without leaving the session.
- Submission flows through `submit_issue`, threaded follow-up runs through `add_issue_message`, and the reporter's agent reads progress through `get_issue_status` with a guest read-back token. Reporter environment (git SHA or app version) is schema-enforced so reports are reproducible.
- On the maintainer side, a triage daemon subscribes to issue creation events through substrate signaling, mirrors to GitHub when warranted, opens a worktree, runs an agent session, and either resolves the bug or asks for clarification through the same thread.
- This is the cleanest example yet of the nervous system framing in action: the substrate signals on write, consumers decide what to do. The substrate does not prioritize, route, or retry — those are operational-layer concerns.
- Both halves of the loop run as agents talking to agents over one MCP surface. The human never opens a browser tab to evaluate, file, or check status. The translation step that historically killed feedback loops has been absorbed into tool calls.
- Next: agent-side subscriptions so reporters get pushed updates instead of pulling, generalizing guest-submission policies to other entity types, and bidirectional peer sync of issue threads across instances.
