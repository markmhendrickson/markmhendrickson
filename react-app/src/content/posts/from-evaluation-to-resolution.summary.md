- Agent-led evaluation via neotoma.io/evaluate told users whether the product fit. The new issues subsystem in v0.12.0, v0.12.1, and v0.13.0 closes the other half of the loop: when the user's agent hits a bug, it files the report through MCP without leaving the session.
- As of v0.13.0 the MCP instructions tell agents to file via `submit_issue` immediately when a reportable problem is confirmed, without prompting for consent. Asking "should I file this?" was the last friction step; it is gone.
- Submission flows through `submit_issue` with a schema-enforced reporter environment (git SHA or app version, the latter auto-populated when omitted). The agent can pass `entity_ids_to_link` so the issue lands wired into the graph it is about — `REFERS_TO` relationships are created server-side in the same atomic call.
- Threaded follow-up runs through `add_issue_message`, with `remote_submission_error` on the response to handle the partial-success case where GitHub accepts but local append fails. The reporter's agent reads progress via `get_issue_status` using a guest read-back token.
- On the maintainer side, a triage daemon subscribes to issue creation events through substrate signaling, mirrors to GitHub via `sync_issues --push` with PII redaction at the boundary, opens a worktree, runs an agent session, and either resolves the bug or asks for clarification through the same thread.
- v0.13.0 lets entities carry their own `schema_instructions` and `entity_instructions`, surfaced on retrieval, so issues can transmit behavioral context to whichever agent reads them next — without crossing the line into substrate orchestration.
- This is the cleanest example yet of the nervous system framing in action: the substrate signals on write, consumers decide what to do. The substrate does not prioritize, route, or retry — those are operational-layer concerns.
- Both halves of the loop run as agents talking to agents over one MCP surface. The human never opens a browser tab to evaluate, file, or check status.
- Next: agent-side subscriptions so reporters get pushed updates instead of pulling, generalizing guest-submission policies to other entity types, and bidirectional peer sync of issue threads across instances.
